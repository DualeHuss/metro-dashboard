About this project

INST 377 final project — University of Maryland, Spring 2025.

About me

I'm Duale Hussein, a junior at UMD studying Information Science. I ride the Metro a lot and always found it annoying that I had to use like three different apps to get all the info I wanted before leaving for the station. That's basically why I built this.

The problem

WMATA already has an API with all this data, it's just not in one place. I wanted to see live arrivals, weather at the station, and know which stations people search for most, all on one page. Took longer than expected honestly but I think it works pretty well now.

Who it's for

  • UMD students commuting into DC (like me)
  • DMV residents who ride the Metro regularly
  • Anyone who doesn't want to dig through the WMATA website

How it works

The frontend is plain HTML, CSS, and JavaScript — no frameworks, which I actually liked because it forced me to understand what was happening. The backend is Node.js + Express. I had to put the API calls on the server side because I couldn't put my API keys directly in the browser JS (they would be visible to anyone). The search history is stored in a Supabase database and that's what powers the bar chart on the home page.

Data sources

  • WMATA API — station list and real-time arrival predictions
  • OpenWeather API — current weather conditions at each station
  • Supabase — stores search history (Postgres database)

Technologies used

  • HTML, CSS, vanilla JavaScript (frontend — no frameworks)
  • Node.js + Express (backend server)
  • Chart.js — JavaScript library used for the "most searched stations" bar chart on the home page
  • Leaflet.js — JavaScript library used for the interactive station map on the trains page
  • Supabase — Postgres database for storing search history
  • Vercel — hosting and deployment

How the frontend connects to the backend

All data on the frontend is loaded through the Fetch API — nothing is hardcoded. There are four fetch calls total:

  • /api/stations — loads the station list for the search dropdown
  • /api/predictions/:code — gets live train arrivals for the selected station (calls WMATA)
  • /api/weather/:code — gets current weather at the station (calls OpenWeather)
  • /api/searches and /api/searches/popular — logs searches to Supabase and loads the chart data

What I'd improve with more time

The popular searches count is being done in JavaScript right now which is kind of inefficient. I'd move that to a SQL view. Also no caching on the predictions endpoint so every search hits the WMATA API directly. For a class project it's fine but it would be a problem at scale.
